Technical issues October 1, 2003 

  • General technical issues:
    • Relationship of new BSML adaptor to existing adaptor
      • Should new adaptor be reverse compatible (e.g. handle Display elements)?
      • Should it handle only TIGR-BSML or also generic BSML?
      • Is it possible to implement a completely generic BSML parser using the BSML object layer?
    • Relationship of objects in BSML object layer to Bioperl objects:
      • BSML objects (e.g. BSML::BsmlSequence) could implement Bioperl interfaces (e.g., SeqI)
      • BSML objects could be converted into Bioperl objects (e.g., RichSeq)
      • Might have problems with the former in dealing with alternatively-spliced BSML transcripts
    • Where should the adaptor code reside?
      • Probably in Bio::SeqIO, Bio::AlignIO, Bio::SearchIO (and Bio::AssemblyIO ?)
      • The Bio::DB package is more appropriate for web-based resources with random access
      • Other classes in Bio::DB likely depend on various *IO packages
    • Will this be part of the Bioperl core?
      • Would add dependency on BSML Perl object layer, so probably not.
      • Although SeqIO::chado* classes appear to have undocumented dependencies

  • Details of the BSML-Bioperl translation:
    • Which Bioperl classes and interfaces should be used?
      • e.g., use of Bio::SeqFeature::Gene classes vs. Bio::SeqFeature::Generic
      • Use of specialized sequence classes to avoid reading sequences from disk
    • How should "mixed-type" documents be handled?
    • How to reconcile Bio::SeqIO::next_seq with BSML CDS->protein HREFs?
      • Read-ahead is the obvious answer
